1.题目

Design and Implementation of Automated IoT Security Testbed 2020

2.主要内容

对以前有关IOT设备安全性的研究进行了全面分析，重点是用于测试IOT设备的各种工具以及已发现的漏洞，此外，本文还包含研究文献中对基于IoT的安全性测试平台的调查。 在本研究中，介绍了一个开放源代码平台，用于识别IoT网络和通信中的弱点。 该平台易于修改和扩展，可以添加新的安全评估测试和功能。 它可以自动执行安全评估，无需人工干预即可进行测试。测试平台报告被测设备的安全问题，并可以检测到针对设备的所有攻击。 它还设计用于监视测试平台内以及与连接的设备之间的通信，从而在检测到恶意活动时使系统中止。 为了演示提议的IoT安全测试平台的功能，它用于检查两个IoT设备的漏洞：无线摄像头和智能灯泡。

2.1论文内容

在将物联网设备推向市场之前，对其进行测试是产品开发中的重要一步，因此测试平台是一个比较有用的领域，安全测试平台是一个预定义的测试环境，在该环境下，所有触发器，测试，攻击和设备都受到控制，据穆拉德等 [12]，由于这些设备的特性和局限性，测试物联网设备可能具有挑战性。 本文的下一部分是全面的文献综述，其中提供了研究，这些研究试图分析IoT设备漏洞并讨论针对每种产品开发的测试。 一些研究人员介绍了用于物联网安全测试平台的结构，但是这些设计中很少有人实施。他们介绍了一种测试平台的结构和实施计划，用于使用封闭源工具作为测试平台乐团来测试IoT设备。 但是，他们的测试平台缺乏可伸缩性，因此很难添加更多测试。 本文的目的是设计仅使用开源工具的综合，易于使用和可重复的自动化IoT安全测试平台。 测试床具有模块化结构，因此可以添加测试而不会影响测试床的结构和行为。

2.2 作者的contribution

  1. 对之前有关IOT设备安全性的研究进行全面分析，阐明在哪些设备使用了哪些工具发现了哪些漏洞
  2. 介绍研究文献中介绍得基于IOT安全性测试平台的调查
  3. 定义用于构建IOT安全测试平台的结构，以使用开源工具评估IOT设备的漏洞
  4. 引入减少用户交互的自动化的测试平台。这将确保对所有链接的设备进行身份验证，以满足安全性要求

3.1关键信息提炼

  1. YE等人对智能锁的研究[24],分析设备漏洞，账户及个人信息暴露情况

  2. Ly对手环进行研究，分析信息泄露的问题[14]

  3. Wurm对智能电表进行研究，提出了解决方案[15]

  4. Tabrizi对智能电表进行研究，提出了分析工具，使用户可以检测恶意活动

  5. KIM对只能门锁进行研究并建议智能锁的应用级别[17]

  6. Chistiakov等 [20]使用电可擦可编程只读存储器（EEPROM）芯片为智能锁开发了一种新的安全设计。 改进的设计包括通过超文本传输协议安全（HTTPS）通道的用户身份验证。

  7. 埃尔南德斯对恒温器进行研究[21]通过USB端口启动恶意映像来测试设备

  8. OREN等人，发现针对智能电视的设备通信协议的攻击[22]

  9. Ur，分析了与智能家居有关的访问控制策略和威胁[26]

  10. Fernandes对Samsung SmartThings进行分析，确定了四种可能针对物联网设备控制APP发起攻击的方法[27]

  11. 西尔 [32]在智能手表上进行了网络分析和固件分析，同时还检查了移动应用程序的漏洞。 作者从IoT设备跟踪了用户的私有地址，捕获了密钥交换，**对移动应用程序进行了反向工程，**监视了该应用程序与Fitbit服务器之间的流量，并使用了代理传输层安全性（TLS）流量来拦截和提取数据。 作者使用了各种工具，包括Ubertooth，Wireshark，crackle，APK Extractor和dex2jar。

  12. [57]专注于评估BLE设备的安全性。 他们使用Wireshark，Kismet和Crackle手动测试了智能手表的安全性。

  13. Sachidananda [54]介绍了一个安全测试平台来分析物联网设备的安全问题。 该测试平台指定了体系结构和设计要求，以支持用于安全性分析的渗透测试的开发。 渗透测试包括端口扫描，指纹识别，进程枚举和漏洞扫描。 他们基于IoT设备市场（即Amazon Echo，Nest Cam，Philips Hue，SENSE Mother，Samsung SmartThings，Withings HOME，WeMo Smart Crock-Pot和Netatmo Security Camera）的安全漏洞进行了测试。 测试平台包括各种IoT设备。但是运用闭源管理软件，可拓展性极差

  14. Hale [55]提出了一个名为SecuWear的开源平台，该平台可以识别商业硬件的漏洞。 SecuWear测试平台可捕获识别不同攻击所需的信息，从而评估可穿戴设备的安全性。 此外，它为安全社区提供了执行攻击和缓解信息的过程。SecuWear的缺点是，仍必须在商用物联网设备中调查Metawear上已识别的漏洞，以确定它们是否适用。 此外，漏洞可能特定于某些开源组件，在将安全问题识别为常见问题时会导致误报。

4.系统设计

在本文中作者提出了一个检测平台，可以评估物联网设备的安全性，对其主要的组成部分以及结构进行了定义。并在无线摄像头（有服务器托管），以及智能电灯（无服务器托管在内网中广播其状态）上进行测试。

4.1 系统功能

  1. 平台基于由开源MS控制的开源工具
  2. 测试平台由接口模块，测试模块，网络模块，报告模块和存储模块组成所有模块可以交互以充当完整的安全测试环节，由MS控制并通过易于使用的GUI向用户显示更新
  3. 测试平台的模块化结构和体系结构允许其他研究人员使用它来构建自己的测试工具。 它是一个灵活且可扩展的系统，这意味着研究人员可以采用初始结构并添加到模块化设计中。
  4. IoT测试平台列出了针对所测试设备以及每个端口中设备托管的服务的所有漏洞利用和CVE。 例如：端口22（SSH）上的OpenSSH 7.6服务。
  5. IoT测试平台会自动生成包含所有设备结果的正式单词报告。

4.2 系统需求

  1. 在测试平台组件之间建立安全的通信
  2. 对网络中所有节点进行身份验证
  3. 控制测试模块和测试顺序
  4. 记录事件和测试结果
  5. 建立可重用的测试和测试平台组件
  6. 确保测试平台的可拓展性

4.3 结构设计

  1. 接口模块：此模块充当I / O接口。 它由两个单元组成：图形用户界面（GUI）单元和输出单元。 GUI从用户那里获取输入提要（需要时），并将其提供给测试模块进行分析。 该方法减少了测试过程中的用户干预。 分析完成后，报告模块将生成摘要报告并将其发送给用户。
  2. 测试模块：此模块管理测试用例并按顺序启动它们。 所有测试用例和脚本都保存在存储模块中。 一旦测试平台投入运行，它就会从数据库中调用通用脚本来检查IoT设备的通用网络特性，根据设备的响应，测试模块将启动更高级的测试用例，以解决安全问题。 例如，在识别出IoT设备中的任何开放端口之后，将启动专用测试用例以测试这些端口的漏洞。 此类漏洞可能包括过时的服务或低安全性配置或身份验证。 此外，将检查IoT设备的响应以确定每个测试用例是否通过
  3. 网络模块：此模块控制网络活动以及与IoT设备的通信。 它创建并监视网络访问点（AP）
  4. 报告模块：此模块生成设备安全评估结果的最终报告。 它是根据测试结果和日志进行编译的
  5. 存储模块：该单元存储由不同模块发起的所有事件，以供以后检索以及评估完成后最终生成报告。 它保存有关已测试设备的所有信息，并存储测试用例清单

4.4 组件设计

  1. DUI：要测试的IOT设备，链接至测试的无线网络
  2. 管理机：运行kali linux的重要组件，使用网络模块审核所有流量包以识别如恶意IP，恶意攻击等内容；同时使用测试模块来启动和协调测试方案，启动测试后管理机分析测试结果以决定下一步进行哪个测试。
  3. 专用无线网络：测试台使用无线路由器模拟网络环境，部署嗅探软件来收集通过本地网络广播的数据。实际上是通过虚拟AP工具出纳兼无线热点搭建本地访问网络，它允许测试平台以管理特权来审核网络流量，从而避免了对ARP欺骗的需求。 虚拟AP工具使用网络模块管理的WPA2-PSK Wi-Fi密钥提供通信加密和安全性。 此外，它还监视DUI的外部连接。
  4. 控制应用程序：测试平台包括一台智能手机设备，用其控制被测物联网设备，可以通过其发送控制消息来检测命令是否以明文发送，检测到攻击后可以复制数据包进行重放攻击。
  5. 攻击机（kali linux）。 攻击机（Kali Linux）用于对DUI发起攻击以发现弱点。 例如，它可以发起重播攻击或暴力密码攻击。 此外，它还会对测试平台和DUI进行DoS攻击，以测试它们的抵抗能力和阻止此类攻击的能力。

4.5 平台的设置情况

作者的实验分两部分进行，半手动的广泛分析阶段（了解IOT设备的通信特性等问题作为第二阶段广泛自动化测试的信息）和自动化测试阶段。

  1. 半自动化阶段：每台设备进行单独测试，使用一系列如zmap等黑客工具,每次测试产生不同的结果，测试是手动完成的。
  2. 自动化分析阶段：模块化的测试结构，每个模块自动运行。

1广泛的分析阶段
  1. 收集信息并扫描漏洞，核心与目前实验室的相似，使用CVE漏洞库对其进行匹配，发现漏洞后进行记录方便DUI进行自动测试评估
  2. 执行Nmap，扫描打开的端口并记录
  3. 检查安全套接字层（SSL）证书。 已测试SSL证书以查看承载Web服务器的DUI是否具有可靠的证书。 可以使用TLS-proper，SSLScan和Nikto工具进行检查。
  4. 检查与时间服务器的异步连接，此测试检查IOT设备是否与时间服务器同步。
  5. 执行HTTP降级攻击： 重点是降低两个节点之间的通信通道中使用的加密级别。 降低安全通道中使用的加密级别可能会导致设备发送信息时根本不进行任何加密，就像HTTP一样。 此测试强制使用SSLStrip，Ettercap，Better-cap等将通信从HTTPS降级为HTTP。如果成功降级了对话，则第三方可能会收集凭据和控制数据包等关键信息。 如果设备拒绝降级并拒绝任何非HTTPS连接，则将其视为安全设备。
  6. 执行凭证和蛮力攻击检查：使用过程中应该强制用户更改默认密码，使用过程中应该限制错误访问试验次数。
  7. 目录暴力攻击：如果DUI托管WEB服务器，为了检查是否有目录没有保护，使用Dirb和DirBuster工具进行暴力攻击。
  8. HTTP绕过验证：防止私有信息泄露或越权访问
  9. XSS和SQL注入：对于具有html界面的托管WEB服务器，可能受到XSS和SQL注入攻击，使用SQLMAP工具进行检查，该工具可以检查HTTP GET请求的参数以注入SQL命令。
  10. 可以对防火墙进行检测。使用WAFWoof工具
  11. 漏洞匹配检查：使用使用Metasploit和Armitage工具检查类似于DUI的设备上的现有漏洞利用程序，觉得也可以使用我们平台的匹配策略。
  12. 分析物联网设备与用户机器之间的通信。拦截物联网设备与机器的通信，检测是否为明文传输等问题。IOTfuzz等现在提出的很多fuzz框架的利用都属于这一步，但是最大的问题是目前大部分需要物理设备
  13. 检查请求的外部IP：在此测试中，测试平台将匹配固件设备的请求IP，与黑名单进行匹配。
  14. 应用程序APP检测，目前很多研究只进行了机器固件本身的分析，动态的fuzz分析，对配套APP的分析较少，所以可以考虑
  15. 固件检测：分析固件中硬编码或者使用命令注入漏洞的存在？？？（看起来是逆向）,此部分可以作为静态分析的一部分。
  16. 物理设备分析：对于未在网络中公开固件的设备，需要拆解等提取（暂时用不到）
2. 自动化测试阶段

根据人工工具分析阶段的测试结果，自动化测试系统将会按算法对每个需要测试的设备进行测试。

  1. IOT设备数量的激增所带来的设备类型和协议种类的繁多，缺乏标准化以及iot设备之间的交互增加了被攻击的概率。作者认为在未来的工作中可以尝试开发一款应对此类多协议问题的测试框架

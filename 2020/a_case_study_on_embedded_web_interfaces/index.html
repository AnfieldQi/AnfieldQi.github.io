1.题目

Automated Dynamic Firmware Analysis at Scale: A Case Study on Embedded Web Interfaces

2.主要内容

2.1 论文内容

提出了第一个完全自动化的框架，它应用动态固件分析技术，以可扩展的方式实现嵌入式固件映像中的自动漏洞发现，介绍了一种方法，并实现了一个可伸缩的框架，用于发现嵌入式web接口中的漏洞，而不考虑供应商、设备或体系结构。为了实现这一目标，框架执行完整的系统仿真，以实现固件映像在仅软件环境中的执行，即不涉及任何物理嵌入式设备。然后使用静态和动态工具分析固件中的web接口。

2.2 作者的contribution
  1. 提出了一个动静态分析结合的自动化测试框架
  2. 介绍了如何动态发现嵌入式WEB接口的框架方法

3.技术背景

  1. 静态分析：不需要依赖过于复杂的模拟环境，静态分析的缺点是FN和FP，二进制静态分析可用于分析CGI等二进制文件以发现漏洞。

  2. 动态分析：web界面的动态分析大多独立于所使用的服务器端技术。例如，同样的工具可以测试用PHP、原生CGIs或自定义web脚本引擎实现的web接口。其次，它可以用来确认在静态分析阶段发现的漏洞。·Arachni[1]、Zed攻击代理（ZAP）[2]和w3af[3]

  3. web界面的运行：固件中70%的问题是在WEB层，对WEB界面进行测试需要web服务正常启动，但是并不是非要整个固件系统都跑起来。

    仿真方式有几种（1）全模拟 （2）重用固件文件中的内核 （3） 进入固件的文件系统，在启动从目录下启动web服务

4. 分析框架设计

4.1 固件的分析过滤

可以直接解包的/包内包含二进制相关文件的/选择具有与WEB服务相关的文件的

4.2 文件系统的准备

解包之后拿到文件系统，确定是否为文件系统的方法是进行相关敏感文件检测，这一步需要对一些解压后的文件系统进行修复，这一阶段主要包含两步：1.文件系统的修复 2. web服务器的存在性探测（可拓展的点：开发工具自动探测相关内容/文中通过目录结构来引导分析，但是现在的会漏掉对动态生成内容的测试）

4.4 结果的收集与分析

在动态和静态分析阶段完成后从安全分析工具中获得分析报告。还收集了一些日志，这些日志可以帮助进一步的分析以及改进框架。这些通常是调试分析工具或我们的仿真环境所必需的。例如，收集与模拟器主机的SSH通信日志、固件文件系统中的更改，并捕获与web接口交互的网络流量。

  1. **文件系统改变：**在以下节点收集快照（i）在开始仿真之前，（ii）在仿真开始之后，以及（iii）在动态分析完成之后，我们在这些快照之间执行文件系统差异。新文件可能是操作系统命令注入的结果，或者更常见的是在动态分析阶段触发的远程代码执行（RCE）漏洞。当动态测试工具尝试插入命令时（例如，touch），通常会发生这种情况。有时，命令注入可以成功，但分析工具无法检测到。但是，使用文件系统diff很容易检测到这种情况
  2. **通信捕获：**执行动态分析涉及（仿真）设备和动态分析工具之间的大量输入和输出数据。捕获通信的原始输入和输出可以增加可信性例如，成功的操作系统命令注入可以不被工具检测到。此外，即使采用“白盒”测试方法（第II-C节），也很难验证此类漏洞。一旦测试阶段结束，就可以发现命令注入实际上是成功的。在这种情况下，需要回滚所有的HTTP事务，以找到触发特定漏洞的输入，然后可以寻找导致此结果的输入和参数（可改进的点：直接使用这部分数据对固件整体进行黑盒测试）
4.5 结果验证

我们手动验证高影响漏洞，并尝试创建一个概念攻击的证据。（近两年已有类似自动化的概念出现，此处可以尝试改进）
